# Логика ассесмента — Sminex People

## Обзор

Ассесмент — это двусторонняя оценка сотрудника:
1. **Само-оценка** — заполняет сотрудник
2. **Оценка руководителя** — заполняет руководитель (только после завершения само-оценки)

Результатом является **грейд** — уровень сотрудника по шкале J1–J3, M1–M3, S1–S3.

---

## Структура компетенций

Каждый шаблон ассесмента содержит набор компетенций.  
Текущий шаблон **Product Owner v2** включает **9 компетенций**:

| # | Код | Название | Блок |
|---|-----|----------|------|
| 1 | PT | Product Thinking & Problem Ownership | Продуктовое мышление |
| 2 | VE | Value & Economics Management | Экономика продукта |
| 3 | BL | Backlog & Prioritization Management | Управление потоком ценности |
| 4 | DL | Delivery & Release Management | Доведение до результата |
| 5 | SC | Stakeholder & Communication Management | Работа с бизнесом |
| 6 | MT | Metrics & Outcome Orientation | Ориентация на результат |
| 7 | PD | Process & Domain Understanding | Понимание предметной области |
| 8 | TW | Product Teamwork & Cooperation | Командная работа |
| 9 | SM | Self-management | Управление собой |

### Структура одной компетенции

```ts
interface Competency {
  id: number
  code: string          // "PT", "VE", ...
  name: string          // полное название
  sub: string           // краткое название блока
  subs: SubCriteria[]   // субкритерии (4–6 штук)
  self_qs: SelfQuestion[] // частотные вопросы для само-оценки
}
```

### Субкритерий

```ts
interface SubCriteria {
  name: string    // название субкритерия

  // Тексты для оценки руководителем (третье лицо):
  j: string       // Junior-уровень
  m: string       // Middle-уровень
  s: string       // Senior-уровень

  // Тексты для само-оценки (первое лицо):
  j_self: string
  m_self: string
  s_self: string
}
```

## Процесс само-оценки (сотрудник)

### Шаг 1 — Выбор уровня по субкритериям

По каждому субкритерию сотрудник видит **3 анонимных варианта** (без подписей Junior/Middle/Senior).  
Каждый вариант — это текст `j_self`, `m_self` или `s_self`.

Выбранное значение сохраняется в `subRatings`:
```json
{ "1-0": 2, "1-1": 1, "1-2": 3, "2-0": 2, ... }
```
Ключ: `"cid-si"` (id компетенции — индекс субкритерия).  
Значение: `1` (Junior), `2` (Middle), `3` (Senior).

### Шаг 2 — Частотные вопросы (`self_qs`)

По каждой компетенции — **3 вопроса** с частотной шкалой:

| Значение | Описание |
|----------|----------|
| 1 | Никогда / крайне редко |
| 2 | Иногда (реже 50%) |
| 3 | Примерно в половине случаев |
| 4 | Часто (>70% случаев) |
| 5 | Почти всегда / всегда |

Сохраняется в `answers`:
```json
{ "1": { "0": 4, "1": 3, "2": 5 }, "2": { ... } }
```

### Шаг 3 — Примеры из практики

По каждому частотному вопросу — необязательное текстовое поле с примером.

## Процесс оценки руководителем

Руководитель видит **те же субкритерии**, но с подписями **Junior / Middle / Senior**.  
Оценивает каждый субкритерий независимо от само-оценки сотрудника.

Добавляет:
- Комментарии по каждой компетенции
- Общий комментарий
- План индивидуального развития (ИПР)

---

## Вычисление грейда

Средний балл по всем субкритериям компетенции:
```
avg = sum(subRatings) / count(subRatings)
```

Функция `getGradeInfo(avg)` в `src/lib/assessment-data.ts`:

| Диапазон avg | Грейд |
|-------------|-------|
| ≥ 2.80 | Сеньор Ранг 3 (S3) |
| ≥ 2.60 | Сеньор Ранг 2 (S2) |
| ≥ 2.40 | Сеньор Ранг 1 (S1) |
| ≥ 2.13 | Мидл Ранг 3 (M3) |
| ≥ 1.87 | Мидл Ранг 2 (M2) |
| ≥ 1.60 | Мидл Ранг 1 (M1) |
| ≥ 1.33 | Джуниор Ранг 3 (J3) |
| ≥ 1.07 | Джуниор Ранг 2 (J2) |
| ≥ 0.80 | Джуниор Ранг 1 (J1) |
| < 0.80 | Ниже Джуниора |

## Gap-анализ

Gap = само-оценка − оценка руководителя (по каждой компетенции).

- **Положительный gap** (`+`) — сотрудник оценивает себя выше руководителя
- **Отрицательный gap** (`−`) — руководитель оценивает выше, чем сотрудник сам себя
- **Нулевой gap** — оценки совпадают

Gap-анализ отображается на странице `/manager/gap/[userId]` как сравнение двух баров по каждой компетенции.

## Жизненный цикл ассесмента

```
Цикл создан (DRAFT)
    │
    ▼
HR выбирает участников → Цикл переходит в ACTIVE
    │
    ▼
Сотрудник заполняет само-оценку (NOT_STARTED → IN_PROGRESS → COMPLETED)
    │
    ▼
Руководитель видит сотрудника в "Ожидают оценки"
    │
    ▼
Руководитель заполняет оценку (NOT_STARTED → COMPLETED)
    │
    ▼
Результаты доступны в /manager/results и /manager/gap/[userId]
    │
    ▼
HR закрывает цикл (ACTIVE → CLOSED)
    │
    ▼
История сохраняется в /employee/history
```

## Текстовые итоги (`getSummaryText`)

После завершения оценки функция `getSummaryText(avg, ratedCount)` возвращает  
автоматический текстовый вывод по грейду:

- Описание текущего уровня
- Что нужно для перехода на следующий ранг
- Рекомендации по развитию

Пример для M3:
> «Мидл Ранг 3 — сильный Middle на пороге Senior. Самостоятельно ведёт продукт, работает с бизнесом и командой. Точечный рост в 2–3 зонах откроет путь к Сеньор Ранг 1.»
