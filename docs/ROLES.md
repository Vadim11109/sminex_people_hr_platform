# Роли и права доступа — Sminex People

## Роли

Платформа поддерживает **3 роли**. Роли хранятся в массиве — пользователь может иметь несколько.

| Роль | Код | Назначается |
|------|-----|-------------|
| Сотрудник | `EMPLOYEE` | По умолчанию при регистрации |
| Руководитель | `MANAGER` | HR вручную через `/hr/users` |
| HR | `HR` | HR вручную через `/hr/users` |

> Роли **не берутся** из Azure AD. Вся ролевая модель управляется HR внутри платформы.

---

## Права доступа по роли

### EMPLOYEE (Сотрудник)

| Действие | Доступ |
|----------|--------|
| Просматривать свои ассесменты | ✓ |
| Заполнять само-оценку | ✓ |
| Видеть свою историю грейдов | ✓ |
| Видеть само-оценки других сотрудников | ✗ |
| Видеть оценку руководителя до завершения цикла | ✗ |
| Изменять роли пользователей | ✗ |

**Доступные разделы:** `/employee/*`

---

### MANAGER (Руководитель)

| Действие | Доступ |
|----------|--------|
| Видеть список своей команды | ✓ |
| Видеть статус само-оценки подчинённых | ✓ |
| Заполнять оценку подчинённых (после их само-оценки) | ✓ |
| Видеть результаты и gap-анализ | ✓ |
| Видеть само-оценки подчинённых | ✓ (после завершения) |
| Управлять циклами или пользователями | ✗ |

**Доступные разделы:** `/manager/*`

---

### HR

| Действие | Доступ |
|----------|--------|
| Назначать/изменять роли пользователей | ✓ |
| Назначать руководителей | ✓ |
| Создавать циклы оценки | ✓ |
| Запускать циклы, выбирать участников | ✓ |
| Закрывать циклы | ✓ |
| Просматривать аналитику по всей компании | ✓ |
| Управлять шаблонами ассесмента | ✓ |
| Управлять системой грейдов | ✓ |

**Доступные разделы:** `/hr/*`

## Пользователи с несколькими ролями

Например, руководитель сам является сотрудником и проходит оценку у своего вышестоящего руководителя.

При нескольких ролях:
- В сайдбаре появляется кнопка **«Сменить режим»** → `/select-role`
- На странице выбора роли пользователь выбирает, в каком режиме работать
- Выбранная роль определяет текущий вид (`/hr`, `/manager` или `/employee`)

---

## Иерархия руководителей

```
Один сотрудник → один руководитель (managerId)
Один руководитель → много подчинённых (reports[])
Руководители не являются субъектами оценки — они оценивают других
```

**Правило:** сотрудник с ролью `EMPLOYEE` + `managerId != null` — может быть включён в цикл оценки.  
Если `managerId == null` — участник недоступен для добавления в цикл.

## Сценарии работы

### Сценарий 1: Первый вход нового сотрудника

1. Сотрудник входит через «Войти через Microsoft»
2. NextAuth создаёт `User` в БД с ролью `[EMPLOYEE]`
3. Сотрудник попадает на `/employee` — пустой дашборд (нет активных циклов)
4. HR заходит в `/hr/users`, назначает руководителя
5. При запуске следующего цикла HR добавляет сотрудника в участники

---

### Сценарий 2: Запуск цикла оценки

1. HR создаёт цикл: `/hr/cycles` → «+ Новый цикл» → заполняет название, даты, шаблон
2. Цикл создаётся со статусом `DRAFT`
3. HR нажимает «Запустить» → открывается модальное окно выбора участников
4. Участники сгруппированы по командам (руководитель + подчинённые)
5. HR выбирает нужных сотрудников (checkbox по отдельному сотруднику или всей команде)
6. Нажимает «Запустить для N» → создаются `Assignment` записи, цикл переходит в `ACTIVE`
7. Сотрудники видят новый цикл в `/employee`

---

### Сценарий 3: Прохождение ассесмента

1. Сотрудник открывает активный цикл → «Продолжить»
2. Заполняет 9 компетенций:
   - По каждому субкритерию выбирает один из 3 анонимных вариантов
   - Отвечает на 3 частотных вопроса (шкала 1–5)
   - Опционально пишет пример из практики
3. Нажимает «Завершить само-оценку»
4. Статус меняется на `COMPLETED`

---

### Сценарий 4: Оценка руководителем

1. Руководитель открывает `/manager/pending`
2. Видит сотрудников со статусом само-оценки «Готово»
3. Нажимает «Оценить →»
4. Заполняет оценку по тем же компетенциям (видит подписи Junior/Middle/Senior)
5. Добавляет комментарии и план развития
6. Нажимает «Завершить оценку»
7. Результаты появляются в `/manager/results` и `/manager/gap/[userId]`

---

### Сценарий 5: Разбор результатов

1. Руководитель и сотрудник встречаются на 1:1
2. Используют gap-анализ `/manager/gap/[userId]` как основу для разговора
3. Обсуждают зоны роста
4. Руководитель при необходимости добавляет ИПР в оценку
5. HR закрывает цикл

| Layout | Роль | Имя |
|--------|------|-----|
| `hr/layout.tsx` | HR, MANAGER, EMPLOYEE | Dev HR |
| `manager/layout.tsx` | MANAGER, EMPLOYEE | Dev Manager |
| `employee/layout.tsx` | EMPLOYEE | Dev Employee |
